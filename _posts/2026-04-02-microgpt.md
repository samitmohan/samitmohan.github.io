> **TL;DR:** MicroGPT is ~340 lines of pure Python that captures the complete algorithm behind GPT: a custom autograd engine, character tokenizer, multi-head causal attention, MLP, and SGD training loop. This post annotates every section from scratch. No frameworks, no magic, no hand-waving.

ChatGPT and the script below share the exact same algorithm. Scale and engineering differ; the math does not. One runs on thousands of H100s trained on the internet. The other runs on your laptop in a few minutes, trained on baby names. The forward pass, the backward pass, the attention mechanism - identical.

This post walks every line. By the end you'll know what each of those 340 lines does and why it's there.

*Prerequisites:* know what a gradient is. If not: read [neural networks](/tech/2025/10/25/nn.html), [the math](/tech/2026/01/21/math.html), and [the PyTorch layer](/tech/2026/01/07/torch.html) first. This builds on all three.

**Core idea:** A language model has one job: predict the next token given the context so far. Everything in this post serves that goal.

## 1. the data (lines 1-23)

The dataset: 32,000 human first names from Karpathy's makemore project. Character-level tokenizer - the alphabet maps to IDs 0-25, and one special token sits at ID 26: `BOS` (beginning-of-sequence). Vocab size = 27.

```python
chars = sorted(set("".join(docs)))   # ['a','b',...,'z'] - 26 chars
BOS = len(chars)                     # ID 26
vocab_size = len(chars) + 1          # 27
```

`BOS` doubles as end-of-name: every name gets wrapped as `[BOS, c1, c2, ..., cn, BOS]`. When the model emits `BOS` during inference, that's the stop signal.

```python
doc = "emma"
tokens = [BOS] + [chars.index(c) for c in doc] + [BOS]
# tokens = [26, 4, 12, 12, 0, 26]

input_tokens  = tokens[:-1]  # [26, 4, 12, 12, 0]
target_tokens = tokens[1:]   # [4, 12, 12, 0, 26]
```

The shift-by-one is the entire supervision signal. At position t, the model sees tokens[0..t] and must predict tokens[t+1]. No labels required - the data labels itself.

BOS doubling as EOS works because the model learns from examples: every name in the training data ends with BOS. After 20 steps it starts producing BOS as the termination signal naturally.

## 2. the autograd engine (lines 25-121)

Training means finding parameters that minimize loss. Gradient descent needs dLoss/d(every parameter). For 4,192 parameters, that's 4,192 separate derivative calculations per step. Autograd computes all of them by tracking every operation and reversing it.

`local_grads` are stored as a plain tuple, not closures. Each operation captures its derivative formula at construction time - no lambda, no deferred computation.

```python
# Python "dunder" methods like __add__ let you define what happens when
# you write a + b: Python calls a.__add__(b) under the hood.
# This means Value objects can participate in normal arithmetic expressions.

class Value:
    def __init__(self, data, _children=(), local_grads=()):
        self.data = data
        self.grad = 0            # dLoss/d(this) - accumulated by backward()
        self._children = _children  # tuple of Value nodes that produced this one
        self.local_grads = local_grads  # d(this)/d(each child) - same order

    # d(a+b)/da = 1, d(a+b)/db = 1
    def __add__(self, other):
        other = other if isinstance(other, Value) else Value(other)
        out = Value(self.data + other.data, _children=(self, other))
        out.local_grads = (1.0, 1.0)   # gradient of sum w.r.t. each input is 1
        return out

    # d(a*b)/da = b, d(a*b)/db = a
    def __mul__(self, other):
        other = other if isinstance(other, Value) else Value(other)
        out = Value(self.data * other.data, _children=(self, other))
        out.local_grads = (other.data, self.data)  # cross-multiply
        return out

    # d(x^n)/dx = n * x^(n-1)
    def __pow__(self, other):
        return Value(self.data**other, (self,), (other * self.data**(other-1),))

    # d(log x)/dx = 1/x
    def log(self):
        return Value(math.log(self.data + 1e-10), (self,), (1/(self.data + 1e-10),))

    # d(exp x)/dx = exp(x)
    def exp(self):
        return Value(math.exp(self.data), (self,), (math.exp(self.data),))

    # d(relu x)/dx = 1 if x>0, else 0
    def relu(self):
        return Value(max(0, self.data), (self,), (float(self.data > 0),))

    # All convenience ops reduce to the primitives above
    def __neg__(self):   return self * -1
    def __sub__(self, other): return self + (-other)
    def __truediv__(self, other): return self * other**-1

    # __radd__, __rmul__ etc. are "reflected" operators.
    # Python calls __radd__(b, a) when a.__add__(b) fails (e.g. int + Value).
    # Without these, `3 * some_value` would raise TypeError.
    def __radd__(self, other): return self + other
    def __rmul__(self, other): return self * other
```

Convenience operators (`__neg__`, `__sub__`, `__truediv__`) add no new gradient rules. They reduce to the three primitives. Backprop traces through them without any extra code.

### backward()

Topological sort first (DFS post-order): every node appears after its children. Then reverse the order - that's the backward order. Seed the root's gradient to 1 (dLoss/dLoss = 1). Walk in reverse: at each node, multiply its `local_grads` by its own `grad` and add to children. That's the chain rule.

`backward()` propagates gradients through the ENTIRE computation graph in one pass - not layer by layer, but through every node that participated in computing the loss.

```python
def backward(self):
    topological_graph = []
    visited = set()

    def build_top(v):
        if v not in visited:
            visited.add(v)
            for child in v._children:
                build_top(child)          # recurse into children first
            topological_graph.append(v)  # append AFTER children = post-order

    build_top(self)
    self.grad = 1                          # dLoss/dLoss = 1
    for v in reversed(topological_graph): # process root before leaves
        for child, local_grad in zip(v._children, v.local_grads):
            child.grad += local_grad * v.grad  # chain rule
```

Three lines to backprop through any computation graph.

One cost: this is scalar autograd. Each number in the network is a separate Python object. A single forward pass on a 16-token input creates roughly 2 million Value nodes. One training step takes ~30 seconds on a laptop. Vectorized PyTorch does the same thing in milliseconds by operating on entire matrices at once. The design here is for reading, not running.

<div class="checkpoint"><strong>Checkpoint: autograd</strong> - You now have a scalar autograd engine. Every arithmetic operation records its local gradient. backward() does one reverse-topological pass and applies the chain rule everywhere. Nothing else is needed.</div>

```python
def softmax(logits):
    max_logit = max(l.data for l in logits)        # stability: subtract max
    exp_logits = [(logit - max_logit).exp() for logit in logits]
    sum_exp = sum(exp_logits)
    return [e / sum_exp for e in exp_logits]

def rmsnorm(x):
    # Normalize by root-mean-square. No learned scale, no mean subtraction.
    eps = 1e-8
    mean_sq = sum(xi**2 for xi in x) / len(x)
    rms = (mean_sq + eps) ** 0.5
    return [xi / rms for xi in x]
```

After softmax, the scores become attention weights. They are non-negative and sum to exactly 1.0 - a probability distribution over the positions the model can attend to.

Normalization stabilizes training by keeping activations in a reasonable range. Without it, residual connections stack slightly-off-scale values layer after layer until gradients explode or vanish.

<div class="surprise"><strong>Design choice:</strong> RMSNorm instead of LayerNorm. Simpler (no mean subtraction, no learned scale/bias parameters). Modern LLMs like LLaMA use RMSNorm for the same reason.</div>

## 5. attention (lines 189-235)

### the problem attention solves

An MLP processes each position independently. Token at position 3 has no access to position 0. But predicting the 'a' in "emm_" is easier with the full context. Attention gives each token a channel to read from every past position.

### Q, K, V

Before computing Q, K, V, ask: who should each token pay attention to? A token representing 'm' in position 2 of 'emma' should probably attend to position 1 ('e') more than position 0 (BOS). Q, K, V give the model the machinery to figure that out from content.

Each token projects its embedding through three learned matrices to produce three vectors:
- **Q** (query): "what am I looking for?"
- **K** (key): "what do I advertise?"
- **V** (value): "what do I carry?"

The canonical formula: $\text{Attention}(Q, K, V) = \text{softmax}\!\left(\dfrac{QK^T}{\sqrt{d_k}}\right)V$

This code implements exactly that, one scalar at a time.

### causal masking

`score(t, s) = dot(Q[t], K[s]) / sqrt(head_dim)`. Dot product measures alignment: if Q[t] and K[s] point in similar directions in 16-dimensional space, their dot product is large. Large score means high attention weight, which means s's value gets pulled into t's output. This is content-based routing: the model routes information based on what the query is asking and what the key is advertising.

The `sqrt(head_dim)` scaling keeps dot products from growing proportionally to dimensionality. Without the sqrt(head_dim) scaling, dot products grow with the embedding dimension. At high dimension, a few scores dominate, softmax saturates (one weight goes to 1.0, rest go to 0), and gradients vanish. The scaling prevents this.

Without the causal mask, every position can see all future tokens - the model learns to copy from the future rather than predict it. Remove `range(t+1)` and replace with `range(len(x_seq))` and the model immediately hits near-zero loss on training data while failing completely on generation.

Causal mask: the loop runs `s in range(t+1)`, so future tokens never exist in the score list. No explicit mask matrix, no -inf padding - just don't compute the scores.

This is O(T²) per head per layer. For T=16: 16×16=256 dot products per head, 1024 total. Fine here. At T=100K (long-context models), this is why flash attention matters.

### multi-head

Split Q, K, V into `n_head=4` slices of `head_dim=4` each. Run attention independently per head. Heads specialize: one might track character repetition ("mm" in "emma"), another common name endings, another positional distance. Nobody tells them to - it emerges from training. Concatenate the 4 outputs back to 16 dims, then project through `wo`.

The output of each attention head is a weighted average of value vectors: head_out[d] = sum over s of weights[s] * V[s][d]. That's it. Attention is just a learnable weighted sum.

### the residual connection

After attention: `x = x + attn_out`. Skip connection from ResNet. Without it, gradients multiply through every layer and hit zero before reaching early parameters. With it, the gradient path from loss to input includes a direct addition - the derivative of `x + f(x)` w.r.t. `x` is `1 + df/dx`, so the gradient is never zero even if `f` learns nothing. See [attention-residuals](/tech/2026/03/18/attention-residuals.html).

```python
def attention_block(x_seq, wq_i, wk_i, wv_i, wo_i):
    Q = [linear(x, wq_i) for x in x_seq]  # T x n_embed
    K = [linear(x, wk_i) for x in x_seq]
    V = [linear(x, wv_i) for x in x_seq]

    out_seq = []
    for t in range(len(x_seq)):
        x_attn = []
        for h in range(n_head):
            start, end = h * head_dim, (h + 1) * head_dim
            qh = Q[t][start:end]

            scores = []
            for s in range(t + 1):  # causal: no peeking at future
                score = sum(qi * ki for qi, ki in zip(qh, K[s][start:end]))
                score = score / math.sqrt(head_dim)  # scale to stabilize softmax
                scores.append(score)

            weights = softmax(scores)

            head_out = []
            for d in range(head_dim):
                # weighted sum of values across attended positions
                val = sum(weights[s] * V[s][start + d] for s in range(t + 1))
                head_out.append(val)
            x_attn.extend(head_out)

        out_seq.append(linear(x_attn, wo_i))  # mix across heads
    return out_seq
```

<div class="checkpoint"><strong>Checkpoint: attention</strong> - Attention computes content-based weighted averages of value vectors. Multi-head runs this in parallel across different learned projections. Causal masking prevents looking at future tokens. The math is three linear projections, a scaled dot product, a softmax, and a weighted sum.</div>

## 6. the MLP (lines 238-248)

Attention mixes information across positions. MLP processes each position's vector independently. Expand 16 → 64 (4x width), apply ReLU, project back 64 → 16. The expansion gives each position capacity to compute non-linear functions of its representation. In larger models this is where most of the "knowledge" is thought to live - factual associations, syntactic patterns.

MLP = per-token compute. Attention = cross-token communication. The transformer alternates between these two operations: attention lets tokens talk to each other, MLP lets each token process what it heard.

In larger models, the MLP layers are where most factual associations are stored - things like 'anna' implies female name. Attention routes information; MLP transforms it.

<div class="surprise"><strong>Design choice:</strong> ReLU instead of GELU. GELU is smoother and used in GPT-2/GPT-3. ReLU is simpler and works fine here.</div>

```python
def mlp_block(x, w_fc1_i, w_fc2_i):
    x = linear(x, w_fc1_i)          # 16 -> 64: project to wider space
    x = [xi.relu() for xi in x]     # non-linearity (ReLU, not GELU)
    x = linear(x, w_fc2_i)          # 64 -> 16: project back
    return x
```

## 7. the full forward pass (lines 251-271)

Positional embeddings are not optional. The attention mechanism is permutation-equivariant: without position information, 'emma' and 'mame' produce identical representations because attention treats the set of tokens the same regardless of order. wpe encodes order so the model can distinguish position 0 from position 3.

```python
def gpt(tokens):
    # 1. Build input embeddings: add token embedding + position embedding
    # enumerate(tokens) gives (0, tok0), (1, tok1), ...
    # wte[tok] is a 16-dim list; wpe[pos] is a 16-dim list
    # zip pairs them element-by-element and adds them
    x_seq = [[te + pe for te, pe in zip(wte[tok], wpe[pos])]
             for pos, tok in enumerate(tokens)]
    # x_seq is now a list of T vectors, each of length 16

    for i in range(n_layer):
        # 2. Normalize, run attention, add residual (x = x + attn_out)
        normed = [rmsnorm(x) for x in x_seq]
        attn_out = attention_block(normed, wq[i], wk[i], wv[i], wo[i])
        x_seq = [[a + b for a, b in zip(attn, x)]   # add skip connection
                 for attn, x in zip(attn_out, x_seq)]
        # ^ for each position: new_x[d] = attn_out[d] + old_x[d]

        # 3. Normalize, run MLP, add residual (same pattern)
        x_seq = [[a + b for a, b in zip(mlp_block(rmsnorm(x), w_fc1[i], w_fc2[i]), x)]
                 for x in x_seq]

    # 4. Project each position's 16-dim vector to 27 logits (one per vocab token)
    return [linear(x, lm_head) for x in x_seq]
```

**Pre-norm** (RMSNorm before each sub-layer) vs post-norm (after): pre-norm keeps the residual stream unmodified by the norm. The original Transformer used post-norm; GPT-2 switched to pre-norm after finding it more stable at scale. LLaMA kept pre-norm with RMSNorm.

Residual connections let the model start from identity (pass x through unchanged) and learn deltas. The derivative of x + f(x) with respect to x is 1 + df/dx - the gradient always has a clean path back through the 1 term. Without residuals, gradients vanish through many layers.

The model is end-to-end differentiable. Every operation - embeddings, attention scores, softmax, weighted sum, MLP, final projection - is composed of Value nodes. loss.backward() computes dLoss/d(every weight) in one pass.

## 8. training loop (lines 274-318)

### loss: cross-entropy

Cross-entropy loss = -log(p[target]). If the model assigns 90% probability to the correct token: loss = -log(0.9) = 0.11. If it assigns 1%: loss = -log(0.01) = 4.6. The log function penalizes confident wrong answers harshly and rewards confident correct ones. It's the only loss function with that property that's computationally stable.

Why log? Cross-entropy is the negative log-likelihood of the data under the model. Maximizing log-likelihood is equivalent to minimizing KL divergence to the true distribution. Practically: log turns products of probabilities into sums, numerically stable and easier to differentiate.

### backward + SGD

```python
for step in range(num_steps):
    total_loss = Value(0.0)

    for b in range(batch_size):
        # modulo wraps around so we cycle through all docs repeatedly
        doc = docs[(step * batch_size + b) % len(docs)]
        # chars.index(c) converts a character to its integer ID
        tokens = [BOS] + [chars.index(c) for c in doc] + [BOS]
        tokens = tokens[:block_size + 1]    # truncate if name > 16 chars

        input_tokens  = tokens[:-1]         # model sees this
        target_tokens = tokens[1:]          # model must predict this

        logits_seq = gpt(input_tokens)

        loss = Value(0.0)
        for logits, target in zip(logits_seq, target_tokens):
            probs = softmax(logits)
            loss = loss + (-probs[target].log())   # negative log-likelihood
        total_loss = total_loss + loss / len(logits_seq)

    avg_loss = total_loss / batch_size
    avg_loss.backward()          # backprop through the ENTIRE Value graph

    # SGD: param -= lr * grad
    for param in params:
        param.data -= learning_rate * param.grad
        param.grad = 0           # must reset or gradients accumulate

    print(f"Step {step+1}/{num_steps}, Loss: {avg_loss.data:.4f}")
```

Gradients accumulate by += in backward(). If you skip the zero step, gradients from step 1 add to gradients from step 2, tripling the effective learning rate and causing divergence. Zero every step.

<div class="surprise"><strong>Surprise:</strong> this is vanilla SGD, not Adam. Three lines: subtract gradient, zero gradient, move on. Adam tracks per-parameter momentum and variance, which matters at scale but adds 2x memory and complexity. For 4K parameters and 20 steps, SGD works. The code comment says "SGD update" explicitly - if you expected Adam, look at the actual lines.</div>

SGD works fine at this scale. The model has 4,192 parameters, a single layer, and 20 training steps. Adam's adaptive learning rates help at scale (millions of parameters, hundreds of layers, noisy gradients) but add two extra moment terms per parameter. For a demo that fits on one page, SGD is correct.

Loss starts at ~3.3 (-log(1/27) ≈ 3.3, random model with 27 classes). It drops to ~2.37 over 20 steps. The model has seen 160 names (20 steps × 8 batch). Output at this point: names are phonetically plausible but rough - "aria", "eman", "marn". Train for 500 steps and they get much better.

## 9. inference (lines 321-343)

Start from `[BOS]`. Run the full forward pass. Take logits at the last position. Apply temperature. Softmax. Sample. Append. Repeat until BOS comes back out.

Logits are unnormalized scores before softmax. A logit of 5.0 doesn't mean 'probability 5.0' - it just means this token is more likely than one with logit 2.0. Softmax turns the relative differences into a probability distribution.

That's all autoregressive generation is: run the model, get probabilities over next tokens, pick one, condition on it, repeat. The model never "sees" what it's about to generate - it only sees what it has generated so far.

**Temperature** is `logit / T` before softmax. If the model's top logit is 2.0 and second is 1.5:
- T=0.5: logits become 4.0 and 3.0 - gap widens, top token dominates
- T=1.0: unchanged
- T=2.0: logits become 1.0 and 0.75 - gap narrows, more even distribution

Temperature controls the entropy of the output distribution. T=0 gives argmax (always pick the most likely token, no diversity). T=1 samples from the raw model distribution. T>1 flattens the distribution (more surprising outputs). Sampling instead of argmax gives diversity: two runs with the same context produce different names.

T=0.5 produces repetitive but valid-looking names. T=1.5 produces novel but sometimes nonsensical ones. The default here is T=0.5.

Attention weights depend on the content of Q and K, not their positions. Position information enters only through the initial wpe embeddings. Once inside the transformer, everything is content-based.

## what to ignore, what matters

**Ignore for understanding the algorithm:**
- The specific architecture choices (ReLU vs GELU, RMSNorm vs LayerNorm) - these are swappable hyperparameters
- The specific numbers (n_embed=16, n_head=4) - scaled up this becomes GPT-3
- The optimizer (SGD vs Adam) - different update rule, same learning signal
- Performance (this is slow) - pure Python with scalar autograd, ~30s/step

**What matters:**
- The autograd engine: any differentiable computation can be trained
- Token + position embeddings encode what and where
- Attention: content-based weighted averaging across positions (causal)
- MLP: per-position nonlinear transformation
- Residual connections: gradient highway through depth
- Cross-entropy loss: the supervision signal
- The loop: forward, backward, update, zero grads

| ReLU | GELU |
| RMSNorm | LayerNorm (GPT-2) / RMSNorm (LLaMA) |

Same algorithm. Scale and engineering differ.

The core attention mechanism is from Vaswani et al. 2017. It hasn't changed. What changed between 2017 and GPT-4 is data pipeline, post-training (SFT + RLHF), infrastructure, and compute budget. The 340 lines above contain the math. The other 99% is systems engineering.
